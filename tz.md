# ТЗ: silver.education_population_wide

## Контекст и предыстория

В текущем решении уже реализованы bronze, bronzenormalized и набор silver-таблиц
по отдельным доменам:
- `silver.doshkolka`
- `silver.naselenie`
- `silver.oo`
- `silver.spo`
- `silver.vpo`
- `silver.dpo`

Образовательные данные изначально пришли из нескольких PostgreSQL-источников:
`oo1271209`, `oo1272211`, `1210143`, `121354`, `124180`.

Для них была реализована нормализация уровня образования через
`bronzenormalized.educationlevel`, где уровень определяется по-разному в каждом
источнике:
- где-то по `columnname`
- где-то по `columnmetadata1`
- где-то по `columnmetadata2`
- где-то по `rowname`

Именно поэтому в silver уже появились раздельные доменные таблицы. Теперь
следующий шаг — собрать их в одну технически согласованную wide-таблицу с
населением для последующего gold-сопоставления.

---

## Предыстория: что суммировалось на предыдущем silver-этапе

### `silver.oo`
Собирается из двух источников `oo1271209` и `oo1272211` через UNION и SUM:
- уровни `1.2`, `1.3`, `1.4`
- значения суммируются по `regioncode + year + age + levelcode`
- до этого в источниках были разрезы по форме обучения (очно / очно-заочно /
  заочно) и по типу учреждения (государственное / негосударственное)
- в `silver.oo` хранится уже итоговая сумма по каждому уровню

### `silver.spo`
Собирается из `1210143`:
- программы `2.5.1` (на базе основного общего) и `2.5.2` (на базе среднего
  общего)
- значения суммируются по `regioncode + year + age + levelcode + programcode`
- до этого в источнике были разрезы по форме обучения и по типу учреждения
- в `silver.spo` хранится итоговая сумма по каждой программе

### `silver.vpo`
Собирается из `121354`:
- уровни `2.6`, `2.7`, `2.8`
- значения суммируются по `regioncode + year + age + levelcode`
- в источнике были разрезы по форме обучения и по аттестации экстернов
- в `silver.vpo` хранится итоговая сумма по каждому уровню

### `silver.dpo`
Собирается из `124180`:
- программы `4.8b.1` и `4.8b.2`
- возраст закодирован в `columnname`, а не в `rowname` — это особенность данного
  источника
- в `silver.dpo` хранится значение по каждой программе

### `silver.doshkolka`
Собирается из Excel-источника через `bronze.exceltables`.
Содержит разрезы:
- `territorytype`: `total`, `urban`, `rural`
- `agegroup`: `total`, `age0`, `age1`, ..., `age7plus`
- `value`

Дополнительная агрегация не требуется.

### `silver.naselenie`
Собирается из Excel-источника через `bronze.exceltables`.
Содержит показатели:
- `totalboth`, `totalmale`, `totalfemale`
- `urbanboth`, `urbanmale`, `urbanfemale`
- `ruralboth`, `ruralmale`, `ruralfemale`

## Итоговая схема

**Имя таблицы:** `silver.education_population_wide`

**Ключ строки** (уникальный): `regioncode + year + age`

| Колонка | Тип | Источник |
|---|---|---|
| `regioncode` | string | ключ |
| `regionnameraw` | string | из любого источника |
| `year` | int | ключ |
| `age` | string | ключ |
| `population_total` | int | `silver.naselenie.totalboth` |
| `level_1_1` | int | `silver.doshkolka`, только `territorytype=total` |
| `level_1_2` | int | `silver.oo` |
| `level_1_3` | int | `silver.oo` |
| `level_1_4` | int | `silver.oo` |
| `level_2_5_1` | int | `silver.spo` |
| `level_2_5_2` | int | `silver.spo` |
| `level_2_6` | int | `silver.vpo` |
| `level_2_7` | int | `silver.vpo` |
| `level_2_8` | int | `silver.vpo` |
| `level_4_8b_1` | int | `silver.dpo` |
| `level_4_8b_2` | int | `silver.dpo` |

Отсутствующие значения хранятся как `NULL` — ожидаемо, так как возрастные
покрытия у уровней разные.

---

## Поле `age`

- Хранится как строка
- Принимает одиночный возраст (`"7"`, `"19"`, `"25"`) или диапазон (`"25-29"`,
  `"65+"`)
- Значения приведены к единому виду внутри таблицы
- Сортировка по возрастанию нижней границы диапазона

---

## Правила агрегации

### Город / село
- Если в источнике уже есть готовый total — берётся он
- Если total отсутствует, а есть urban и rural — итог рассчитывается как
  `urban + rural`

### Гос / негос и форма обучения
- В silver уже сложены на предыдущем этапе
- В wide-таблицу попадает итоговая сумма

### Дошколка
- Берётся только строка с `territorytype = 'total'`
- Разрезы `urban` / `rural` в wide-таблицу не выносятся

---

## Принцип объединения

Объединение по ключу `regioncode + year + age`.

- Строки не должны теряться при отсутствии значения в одном из источников
- Логика — полное объединение всех ключей из всех источников
- Если показатель для строки отсутствует — колонка остаётся `NULL`

## Шаги реализации

### 1. Привести все источники к единому long-виду
Структура промежуточного long-формата:
- `regioncode`
- `regionnameraw`
- `year`
- `age`
- `metric_code`
- `value`

Где `metric_code` принимает значения:
`population_total`, `level_1_1`, `level_1_2`, `level_1_3`, `level_1_4`,
`level_2_5_1`, `level_2_5_2`, `level_2_6`, `level_2_7`, `level_2_8`,
`level_4_8b_1`, `level_4_8b_2`

### 2. Построить полный набор ключей
Собрать все уникальные комбинации `regioncode + year + age` из всех источников.

### 3. Pivot в wide-формат
Развернуть `metric_code` в отдельные колонки.

### 4. Сортировка
Отсортировать по: `regioncode` → `year` → `age`.

### 5. Сохранить результат
Записать в `silver.education_population_wide`.

## Проверки

| Проверка | Ожидание |
|---|---|
| Уникальность `regioncode + year + age` | Нет дублей |
| Отсутствие потери строк при JOIN | Строки не пропадают, появляются NULL |
| NULL в колонках уровней | Допустимо; отражает разные наборы возрастов |
| Диапазон лет | Совпадает с ожидаемым по источникам (2018–2024) |
| Coverage по регионам | Строится контрольная таблица (см. ниже) |
