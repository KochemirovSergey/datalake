# ЧТЗ. Слой 4: Валидация и Покрытие (Data Quality)

## 1. Цель и назначение слоя
Слой 4 предназначен для обеспечения качества данных (Data Quality) перед их загрузкой в эталонные витрины (Слой 5). Основные задачи слоя:
* Аудит полноты данных (покрытие по годам, регионам и уровням образования).
* Предоставление аналитику инструмента для ручной отсечки "рваных" исторических данных (Human-in-the-Loop).
* Сверка контрольных федеральных сумм с официальной статистикой.
* Автоматическая детекция аномальных выбросов в динамике.

Слой реализуется в виде трех параллельных функциональных цепочек для каждой целевой предметной области: Обучающиеся, Общежития, Педагогические кадры.

## 2. Архитектура слоя (Функциональная цепочка)
Каждая предметная ветка разделена на три последовательных логических подслоя (этапа).

### Подслой 4.1: Диагностика (Генерация матрицы покрытия)
На этом этапе формируется сырая статистика заполненности и визуальный артефакт для аналитика.

* **Ассет Data-матрицы (напр., `v_obshagi_coverage_matrix`)**: 
  Принимает данные со Слоя 3.2. Рассчитывает процент заполненности в трехмерном разрезе: `Регион × Год × Уровень образования`. Для таблиц со множеством метрик (например, общежития) расчет производится независимо по каждому параметру (`COUNT(metric_1)`, `COUNT(metric_2)`).
* **Ассет Визуализации (напр., `v_obshagi_heatmap_ui`)**: 
  Генерирует интерактивный HTML-файл с тепловой картой покрытия. Включает dropdown-фильтры для переключения между уровнями образования (ВПО/СПО) и расчетными параметрами. Ссылка на HTML-файл сохраняется в `MetadataValue.url()` текущего запуска Dagster.

### Подслой 4.2: Принятие решений (Фильтрация и разметка)
На этом этапе применяется конфигурация Dagster Config, заполняемая аналитиком на основе тепловой карты.

* **Ассет Фильтрации (напр., `v_obshagi_filter`)**:
  Принимает пользовательский конфиг перед выполнением графа. Осуществляет физическую обрезку данных.

**Параметры Dagster Config:**
* `cut_off_years` (List[int]): Годы, данные за которые подлежат полному удалению из пайплайна.
* `allowed_missing_regions` (List[str]): Перечень регионов (например, новые территории), отсутствие которых игнорируется и не считается ошибкой покрытия.
* `force_allow_incomplete_years` (List[int]): Годы с неполными данными, которые принудительно пропускаются в продакшен. При совпадении строки с этим годом, скрипт добавляет технический флаг `is_incomplete_historical_data = True` для последующей индикации на дашбордах (Слой 7).

### Подслой 4.3: Финальная валидация (Проверка качества очищенных данных)
Этап работает строго с данными, прошедшими через фильтр подслоя 4.2.

* **Ассет Сверки сумм (напр., `v_obshagi_federal_summary`)**:
  Агрегирует данные до федерального уровня (`GROUP BY year, metric`). Формирует таблицу в БД для ручной сверки с официальной статистикой Минпросвещения. Название таблицы в метаданные ассета.
* **Ассет Детекции аномалий (напр., `v_obshagi_anomaly_detector`)**:
  Рассчитывает скользящее среднее и стандартное отклонение по каждому показателю в разрезе региона. 
  *Правило реагирования:* Если значение выходит за рамки допустимого отклонения в 1 или 2 точках (годах), генерируется `Warning` в логах Dagster и строка добавляется в журнал выбросов. Если отклонение фиксируется 3 и более лет подряд, это трактуется как смена тренда и алерт не формируется.

## 3. Входы и Выходы слоя

**Входящие данные (от Слоя 3.2):**
* Разделенные по таблицам уровни образования (ОО, ВПО, СПО, ПК).
* Данные агрегированы с учетом бизнес-разрезов (гос/негос, город/село, форма обучения).

**Исходящие данные (в Слой 5):**
* Очищенные массивы данных с равномерным покрытием (или с проставленными флагами неполноты `is_incomplete_historical_data`).
* Данные проверены на аномалии (предупреждения зафиксированы).
* Сформированы витрины федеральных сумм для сверки.
