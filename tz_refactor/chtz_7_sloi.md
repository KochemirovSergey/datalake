# ЧТЗ Этап 7: Слой Вывода (Output / Dashboards)

## 1. Задача
Рефакторинг визуализационных скриптов. Превращение дашбордов в "глупые" генераторы статики (stateless renderers) и интеграция с S3.

## 2. Исходное состояние
- Скрипты `generate_heatmap.py`, `dashboards/dormitory_dashboard.py` и `coverage_dashboard.py` содержат сложный `pandas` код (`pd.merge`, `.apply()`).
- Запуск дашбордов слабо связан с успешностью работы ETL-пайплайна.

## 3. Целевое состояние
- Дашборды лишены прав на трансформацию данных.
- Дашборды зарегистрированы как финальные узлы (Downstream Assets) в Dagster и запускаются строго после успешного обновления схемы `metrics` или `validation`.
- Результат работы (HTML файлы) автоматически улетает в S3.

## 4. Требования к реализации

### 4.1. Выпиливание математики
Полностью удалить все вычисления из фронтенд-скриптов. 
Запрос к базе должен выглядеть как `SELECT region_name, year, indicator_value, status_color FROM metrics.m_target_view`.

### 4.2. Использование готовых цветов
При генерации CSS для ячеек таблиц или тепловых карт использовать значения из колонки `status_color`, которую посчитал Слой 6. Запрещено использовать функции вида `def get_color(val):` внутри скриптов дашбордов.

### 4.3. Выгрузка в S3
После того как скрипт сгенерировал `.html` файл в локальную файловую систему, он должен вызвать метод из `utils/s3_utils.py` (загрузить файл в бакет). Узел Dagster при этом может ничего не возвращать в БД (`return None` или возвращать объект `dagster.AssetMaterialization` со ссылкой на S3).

## 5. Требуемые ассеты (dagster_project/assets/7_output_assets.py)
Группа `7_dashboards` (Без сохранения в БД):
- `dash_dormitory` (ждет `m_dormitory_shortage_heatmap`)
- `dash_staff` (ждет `m_staff_shortage_index`)
- `dash_coverage` (ждет `v_coverage_matrix` из Слоя 4)

**Конфигурация:** Использовать стандартный IO Manager или отключить его для этой группы.
