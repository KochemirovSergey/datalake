# ЧТЗ Этап 4: Слой Валидации (Validation)

## 1. Задача
Создание единой матрицы покрытия (`v_coverage_matrix`). Полный отказ от разрозненных скриптов проверок (asserts) в пользу табличного (декларативного) контроля качества.

## 2. Исходное состояние
- В папке `validation/` находится множество скриптов (`validate_silver.py`, `validate_dormitory.py` и т.д.), которые проверяют аномалии и выбрасывают ошибки.
- Логика проверки полноты данных ("дыр" по годам/регионам) зашита прямо в код дашборда `coverage_dashboard.py`.

## 3. Целевое состояние
- Все старые скрипты из папки `validation/` **удаляются или игнорируются** агентом. Мы перенесли бизнес-очистку на Этап 2.
- Создается **ровно один** новый узел (Asset), который строит широкую сводную таблицу наличия данных.

## 4. Требования к реализации (Логика `v_coverage_matrix`)

### 4.1. Генерация эталонной сетки
Узел должен принимать на вход справочник регионов (из 1 уровня) и список лет (из конфига или жестко заданный, например, 2018-2024).
Необходимо построить декартово произведение (Cartesian Product): `Все валидные регионы x Все года`.

### 4.2. Сборка флагов наличия (Coverage)
К эталонной сетке нужно сделать `LEFT JOIN` уникальных связок `[region_id, year]` из всех фактовых таблиц Уровня 3 (`a_doshkolka`, `a_naselenie`, `a_obuch_vpo` и т.д.).
Для каждого источника сформировать булеву колонку (например, `has_doshkolka`, `has_vpo`), которая принимает `True`, если данные есть, и `False`, если при джойне получился `NULL`.

## 5. Требуемые ассеты (dagster_project/assets/4_validation_assets.py)
Группа `4_validation`:
- `v_coverage_matrix` (на вход принимает `regions` и все `a_*` таблицы Уровня 3).

**Конфигурация:** Настроить IO Manager на сохранение в схему `validation`.
