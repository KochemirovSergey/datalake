# ТЗ: Синхронизация Gold-слоя и генерация Knowledge Graph (KG.md)

## 1. Общее описание и архитектура
Новый модуль отвечает за финальный этап пайплайна: доставку готовых аналитических витрин из локального Data Lake (DuckDB) в целевую базу данных PostgreSQL (`kg_db`) и автоматическую актуализацию примеров данных в графе знаний (`KG.md`).

Процесс запускается администратором вручную и состоит из двух последовательных шагов:
1. Полное пересоздание базы/таблиц в PostgreSQL и заливка свежих данных из Gold-слоя.
2. Парсинг шаблона `KG.md` и вставка актуальных срезов данных и SQL-скриптов.

## 2. Шаг 1: Миграция данных (DuckDB → PostgreSQL)
* **Инструмент:** Расширение `postgres_scanner` в DuckDB.
* **Источники (DuckDB):** `gold.students`, `gold.staff_load`, `gold.dormitory`.
* **Целевая БД:** PostgreSQL (`kg_db`).
* **Логика обновления (Clean Slate):** * При каждом запуске скрипт должен **полностью зачищать старые данные**. 
  * Рекомендуемый подход: `DROP SCHEMA public CASCADE; CREATE SCHEMA public;` (или жесткий `DROP TABLE IF EXISTS` для каждой таблицы), чтобы гарантировать отсутствие артефактов от предыдущих запусков.
  * После очистки DuckDB выполняет копирование таблиц "как есть": `CREATE TABLE pg_db.table_name AS SELECT * FROM gold.table_name;`.
  * Возникающий при этом даунтайм целевой базы данных допускается (система предназначена для внутренних нужд).

## 3. Шаг 2: Логика парсинга и обновления `KG.md`
* Скрипт-генератор читает существующий файл `KG.md`.
* Скрипт осуществляет поиск маркеров — заголовков любого уровня вложенности (`#`, `##`, `###` и т.д.), содержащих строгий паттерн: `Пример таблицы \`имя_таблицы\``.
* При нахождении маркера скрипт **удаляет всё содержимое** от этого заголовка до следующего markdown-заголовка (строки, начинающейся с `#`) или до конца файла.
* Весь остальной текст в файле (ручные описания, таксономия, комментарии) должен остаться нетронутым.
* На освободившееся место под маркером вставляется сгенерированный скриптом блок (см. раздел 4).

## 4. Структура генерируемого блока (3 секции)
Для каждой из трех таблиц генерируется строго 3 логические секции.

### Секция 1: Федеральные агрегаты (RU-FED) + SQL
Поскольку значения `RU-FED` физически не хранятся в Gold-таблицах, скрипт должен высчитывать их на лету SQL-запросом к обновленному PostgreSQL.
* **Обязательное условие:** Перед таблицей с результатами должен выводиться Markdown-блок ````sql ... ````, содержащий реальный запрос, которым эти федеральные цифры были получены. Это служит инструкцией для LLM-агента.
* **Схема колонок:** Результат агрегации должен повторять структуру колонок оригинальной таблицы (насколько это применимо).
* **Специфика расчетов по таблицам:**
  1. **`gold_students` (Обучающиеся):** * Фильтр: только **последний доступный год**. 
     * Расчет: вычисляется процент охвата (отношение количества обучающихся к численности населения соответствующего возраста — данные берутся из этой же таблицы).
     * Вывод: **ТОП-20** возрастов с самым высоким процентом покрытия.
  2. **`gold_staff_load` (Кадровое обеспечение):** * Фильтр: за **3 последних доступных года**.
     * Расчет: агрегаты в разрезе уровней образования.
  3. **`gold_dormitory` (Инфраструктура общежитий):** * Фильтр: за **все доступные годы**.
     * Расчет: агрегация площадей и обеспеченности местами по стране в целом.

### Секция 2: Примеры реальных строк
* Скрипт извлекает из базы **2 случайные строки** регионального уровня (не федерального).
* Данные форматируются в стандартную Markdown-таблицу.

### Секция 3: Словарь возможных значений
* Скрипт собирает список уникальных значений для каждой колонки таблицы.
* **Лимиты:** * Если уникальных значений `<= 20`, выводятся все. 
  * Если `> 20`, выводится 20 первых/случайных значений, а последней строкой добавляется пометка `... (N всего)`, где N — общее число уникальных значений.
* **Формат:** Таблица, в которой каждая колонка содержит перечень значений друг под другом (как реализовано в текущей версии `KG.md`).

## 5. Каркас (шаблон) файла KG.md
Администратор создает и поддерживает следующую базовую структуру файла. Скрипт модифицирует только зоны под заголовками `Пример таблицы...`.

```markdown
# Граф знаний о системе образования РФ

[Ручное описание онтологии, таксономии и связей...]

## Аналитические витрины (Gold Layer)

### Обучающиеся (Students)
[Описание структуры таблицы обучающихся...]
